# aeDaemon: Руководство разработчика   ## Назначение системы Автоматизация производства графики в Adobe After Effects в рамках концепции Graphics-as-a-Service.   ## Архитектура проекта Система представляет собой сборочный воркер (**Automation Worker**), работающий совместно с сервером задач, и состоит из следующих компонентов. ### 0. Сервер задач (Task Broker): Внешний компонент (HTTPS), спроектирован по паттерну очереди задач (Task Queue), выполняет следующие функции: - Регистрация и валидация заказов - Управление очередью (Queue Management) - Хранение состояний (State Machine) - Предоставление данных через опрос очереди (Pull Model) ### 1. Скрипт обвязки (Automation Wrapper) Ядро системы (ExtendScript + AE), реализует основные процессы автоматизации (Automation Pipeline): #### Инфраструктурные функции: - Загрузка конфигураций - Отрисовка интерфейса (UI) - Ведение логов (Logging) #### Взаимодействие с сервером задач: - Опрос очереди (Task Polling) и получение параметров задачи - Контроль и изменение статусов задачи на сервере #### Управление процессами After Effects: - Загрузка шаблона AE - Вызов соответствующего модуля сборки - Формирование очереди рендера AE - Сохранение проекта #### Управление рендерингом: - Отправка на рендер (Render Dispatch) - Мониторинг и локальное журналирование состояний (Local State Logging) ### 2. Модуль сборки (Automation Builder) Исполнительный модуль (ExtendScript + AE), осуществляет подготовку проекта AE на базе выбранного шаблона. Выполняет комплексные действия, руководствуясь специализированным внутрисистемным форматом данных (Custom Data Format): - Поиск и валидация композиций - Преобразование форматов данных (Data Transformation) - Композитинг и наложение спецэффектов (Compositing) - Динамическое наполнение проекта (Asset & Data Injection) - Последовательный монтаж (Media Sequencing) - Модификация нативных параметров слоя - Регистрация сообщений об ошибках (Error Logging) ### 3. Супервизор (Automation Operator) Сервисный модуль (Node.js), выполняет сопутствующие функции: - Мониторинг и локальное журналирование состояний в непрерывном режиме - Синхронизация журналов и статусов на сервере - Транскодирование медиафайлов (FFmpeg) - Выгрузка результатов в облако (S3 Storage)   ## Методы обеспечения надежности - Самовосстановление после сбоев (Crash Recovery) На узле исполнения (Worker Node) ядро запускается в скрытом (Silent) режиме с помощью системного планировщика (Windows Task Scheduler) периодически с фиксированным интервалом. Перед каждым запуском выполняется процедура принудительного завершения процессов `afterfx.exe`. Такой подход исключает неконтролируемое потребление ресурсов и потерю связи с приложением. - Идемпотентность сборки (Idempotency) Пайплайн исполняется циклически до исчерпания задач в очереди. В случае вынужденной остановки или перезагрузки прерванная задача берется в обработку повторно при следующем запуске до исчерпания лимита попыток. Все файлы перезаписываются. - Внешний сетевой транспорт (External Network Transport) Сетевые запросы ядра к серверу выполняются с помощью Curl. Обмен данными происходит посредством временного файла с контролем таймаута. Такое решение позволяет более гибко и надежно управлять сетевым взаимодействием по сравнению с нативным системным вызовом AE. - Поддержка автономного режима (Standalone Mode) Скрипт допускает изолированный ручной (GUI) запуск на рабочей станции пользователя без участия сервисного модуля. - Переключаемый способ рендера (Pluggable Render Engine) - Экспорт в отслеживаемую папку (Watch Folder + AME) - Локальный рендеринг (AE) - Отправка на рендер-ферму (Deadline) - Локальный механизм контроля Журналирование состояний текущих задач позволяет восстанавливать контекст и продолжать мониторинг исполняемых процессов после отказа или перезагрузки.
